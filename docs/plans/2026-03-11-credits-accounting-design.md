# Credits / Учет расходов — дизайн

## Контекст

В приложении уже есть генерация через AI-модели, но еще нет платежной системы и встроенного учета расходов. Пользователи уже работают с продуктом, поэтому нужен внутренний механизм кредитов:

- администратор вручную пополняет баланс пользователя;
- генерации списывают кредиты;
- стоимость зависит от модели и выбранных параметров;
- часть пользователей получает скидку, часть платит по себестоимости, часть с наценкой;
- при ошибке провайдера деньги пользователю возвращаются;
- пользователь заранее видит стоимость и не может запустить генерацию, если средств не хватает.

## Принятые решения

- Все расчеты инкапсулируются в одном доменном контуре `CreditsService`, который работает как банковский счет.
- Формулы запрещены. Стоимость задается только фиксированными lookup-значениями.
- Если параметр не участвует в ценообразовании, его не нужно заносить в прайсинг.
- Модель может иметь:
  - фиксированную стоимость;
  - стоимость по комбинации значимых параметров.
- Пользовательская цена считается как базовая стоимость модели, умноженная на пользовательский процент.
- Процент задается в понятном виде:
  - `100` — по себестоимости;
  - `80` — скидка 20%;
  - `130` — наценка 30%.
- Все пользовательские тексты в UI и API должны быть на русском языке.

## Архитектура

### Domain

- `CreditsService` — единая точка входа для проверки, расчета, списания и возврата.
- `UserBalance` — доменная сущность баланса пользователя.
- `PricingConfig` — описание способа ценообразования модели.
- `CreditsTransaction` — журнал всех операций по балансу.

### Application

Сценарий генерации не должен самостоятельно считать деньги. Он вызывает доменный сервис:

1. посчитать стоимость для текущей модели и параметров;
2. применить пользовательский процент;
3. проверить достаточность средств;
4. при успехе выполнить атомарное списание;
5. при неуспехе вернуть детальную ошибку для UI.

### Integration

- `estimate` используется фронтендом при изменении модели или параметров;
- `debit_for_generation` используется перед запуском генерации;
- `refund_for_generation` используется, если провайдер завершил задачу ошибкой или генерация не состоялась.

## Модель данных

### Пользователь

У пользователя добавляются поля:

- `balance` — текущий баланс в кредитах;
- `pricing_percent` — процент относительно базовой цены модели.

### AI-модель

У AI-модели добавляется `pricing_schema`.

Поддерживаются два режима:

### 1. Фиксированная стоимость

```json
{
  "fixed_cost": "5.00"
}
```

Подходит для апскейлеров и других операций, где параметры не влияют на себестоимость.

### 2. Стоимость по значимым параметрам

```json
{
  "cost_params": ["resolution", "duration"],
  "costs": {
    "720p|5": "3.00",
    "720p|8": "4.00",
    "1080p|5": "5.00",
    "1080p|8": "7.00"
  }
}
```

Ключ в `costs` собирается в порядке `cost_params` через `|`.

Пример для Nano Banana:

```json
{
  "cost_params": ["width", "height"],
  "costs": {
    "512|512": "1.00",
    "768|768": "2.00",
    "1024|1024": "4.00"
  }
}
```

## Правила ценообразования

- Пустой или отсутствующий `pricing_schema` — это ошибка конфигурации, а не бесплатная генерация.
- Если пользователь выбрал комбинацию параметров, которой нет в `costs`, возвращается ошибка: `Не удалось определить стоимость генерации. Обратитесь в поддержку.`
- Все денежные значения в API возвращаются строками, чтобы не терять точность `Decimal`.
- Если `pricing_percent = 0`, генерация для пользователя бесплатна.

## API

### `GET /api/credits/balance/`

Возвращает баланс текущего пользователя и его ценовой процент.

Пример ответа:

```json
{
  "balance": "50.00",
  "pricing_percent": 130,
  "label": "кредиты"
}
```

### `POST /api/credits/estimate/`

Принимает:

```json
{
  "ai_model_id": 1,
  "generation_config": {
    "width": 1024,
    "height": 1024
  }
}
```

Возвращает:

```json
{
  "cost": "5.20",
  "balance": "12.00",
  "can_afford": true,
  "error": null
}
```

Если средств недостаточно:

```json
{
  "cost": "15.00",
  "balance": "3.00",
  "can_afford": false,
  "error": "Недостаточно средств для генерации."
}
```

Если прайсинг сломан:

```json
{
  "cost": null,
  "balance": "3.00",
  "can_afford": false,
  "error": "Не удалось определить стоимость генерации. Обратитесь в поддержку."
}
```

### `POST /api/scenes/{id}/generate/`

Перед созданием `Element` endpoint обязан вызвать списание через `CreditsService`.

Сценарии:

- хватает средств — создается `Element`, стартует Celery-задача;
- не хватает средств — генерация не стартует, UI получает русскую ошибку;
- прайсинг не настроен — генерация не стартует, UI получает русскую ошибку.

## UI

### Navbar

Баланс отображается справа сверху, рядом с переключателем темы и перед аватаром пользователя. Это место выбрано как глобальная и всегда видимая зона.

Формат:

- `50 кр.` для компактного отображения;
- при необходимости позже можно добавить поповер с историей операций, но в текущем дизайне это не требуется.

### ConfigPanel

В левой панели отображается:

- `Стоимость: X кр.`
- ошибка, возвращенная credits-логикой, если стоимость определить нельзя или денег не хватает.

В `ConfigPanel` не показывается баланс, только стоимость и причина недоступности генерации.

### PromptBar

- Кнопка `Сгенерировать` блокируется, если `can_afford = false`.
- Дополнительный текст возле кнопки не нужен.
- Причина блокировки читается в `ConfigPanel`.

## Админка

### Пользователь

В админке пользователя должны быть:

- текущее значение баланса;
- процент ценообразования;
- возможность быстро пополнить или скорректировать баланс;
- история операций.

### AI-модель

В админке AI-модели должны быть:

- `pricing_schema`;
- подсказки с примерами для фиксированной модели и модели с lookup-таблицей;
- явное разделение параметров UI и параметров, влияющих на цену.

## Возвраты и ошибки провайдера

### Когда делаем возврат

Возврат обязателен, если генерация завершилась неуспешно:

- провайдер вернул ошибку;
- сработала content policy;
- задача закончилась failure;
- система не смогла завершить генерацию и перевела элемент в ошибку.

### Почему возврат нужен

Пользователь не должен платить за генерацию, которая фактически не состоялась.

### Что делать с реальным счетом у поставщика

Система должна вести свой учет независимо, но так, чтобы можно было сверять его с реальными расходами провайдера:

- списали пользователю;
- вернули пользователю;
- видим причину и связанный `element_id`;
- можем позже сопоставить это с логами провайдера.

## Журнал операций

Нужна отдельная сущность `CreditsTransaction` с полями:

- `user`;
- `amount`;
- `balance_after`;
- `reason`;
- `element`;
- `metadata`;
- `created_at`.

Поддерживаемые причины:

- `admin_topup`;
- `admin_adjustment`;
- `generation_debit`;
- `generation_refund`;
- `refund_provider_error`;
- `refund_pricing_failure`.

Журнал нужен для:

- аудита;
- поддержки;
- разбирательств по возвратам;
- будущей сверки с расходами поставщика.

## Edge cases

- `pricing_schema` пустой или отсутствует — ошибка `Не удалось определить стоимость генерации. Обратитесь в поддержку.`
- Комбинация параметров не найдена — та же ошибка поддержки.
- Баланс меньше стоимости — генерация недоступна, кнопка заблокирована.
- Поставщик вернул ошибку — выполняется возврат.
- Повторный refund по одной генерации должен быть идемпотентным.

## Требования к надежности

- Списание и изменение баланса должны происходить атомарно.
- Повторная обработка failure не должна приводить к двойному возврату.
- Цена, показанная в `estimate`, и цена, списанная в `generate`, должны считаться одной и той же бизнес-логикой.
- Нельзя дублировать ценообразование в UI.

## Тестирование

Нужны тесты на:

- fixed pricing;
- lookup pricing;
- пользовательский процент;
- недостаток средств;
- отсутствие `pricing_schema`;
- неизвестную комбинацию параметров;
- успешное списание;
- возврат при ошибке провайдера;
- идемпотентность возврата.
