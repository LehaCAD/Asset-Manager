## 2026-04-18 — Регистрация и SMTP

**Симптом:** При регистрации фронт показывает «ошибку запроса», но в админке аккаунт виден — то есть создался. Юзера на `/check-email` не перекидывает.

**Что проверили:**
- Логи `apom_backend`: есть `ERROR apps.users.emails: Failed to send email ... timed out` в момент попытки.
- Логи `apom_nginx`: `POST /api/auth/register/ HTTP/2.0 499 0` — клиент оборвал соединение.
- `EMAIL_TIMEOUT=20`, SMTP-порты на VPS закрыты провайдером (тикет открыт).
- `send_verification_email` реализован через `threading.Thread(...).start()` — fire-and-forget, основной запрос email не ждёт.
- Прямой curl на прод: `POST /api/auth/register/` → `201` за **0.75 с**, токены в ответе есть. То есть SMTP-блокировка **не** задерживает ответ.

**Вывод:** Закрытые SMTP-порты НЕ являются root cause ошибки регистрации. Старая сборка фронта, сетевая задержка в момент попытки, или клиентский timeout (axios 15 с) — но не email. Бэкенд отдаёт 201 быстро и корректно.

**Идея на будущее (defensive UX):**
Если `POST /api/auth/register/` упал по timeout/network error — не показывать юзеру generic «ошибка», а предложить путь вперёд: «Возможно, аккаунт уже создан. Попробуйте войти». Логика: register идемпотентна для одинаковых данных → если повтор даст `400 email уже существует`, значит первый раз прошёл → редирект на `/login` с предзаполненным email.

**Ещё одна идея:** вынести `send_verification_email` в Celery-таск вместо `threading.Thread`. Преимущества — ретраи, видимость в flower, не завязан на живучесть Daphne-процесса. Сейчас, если Daphne перезапустится во время SMTP-попытки, письмо просто теряется.

## 2026-04-18 — Два Dockerfile-а разъехались, libmagic потерян

**Симптом:** Sentry-ошибка `failed to find libmagic`; фото в чате поддержки не отправляются.

**Что проверили:**
- В `backend/Dockerfile` `libmagic1` есть (коммит 72bfb4e, 2026-04-09).
- В рантайм-контейнере `apom_backend`: Python 3.11, Debian trixie, `libmagic` не установлен, пакета нет.
- На сервере `/root/Asset-Manager/docker-compose.production.yml` → `dockerfile: Dockerfile.production` — ДРУГОЙ файл!
- `Dockerfile.production`: Python 3.11-slim, **нет `libmagic1`**.

**Корень:** два параллельных Dockerfile (`Dockerfile` для dev, `Dockerfile.production` для прода) разошлись. При добавлении `libmagic1` изменения попали только в dev-файл. Прод продолжает билдиться без этой зависимости.

**Фикс:** добавил `libmagic1` в `Dockerfile.production`.

**Мысль на будущее:**
- Два Dockerfile-а — ловушка по определению. Любое изменение apt-deps требует синхронной правки двух мест, и ничто не подсказывает, если забыли.
- Кандидаты на рефакторинг:
  - Один Dockerfile с multi-stage + build-arg `ENV=production|dev`.
  - Или — общий base image, от которого наследуются dev и prod.
- Также завести CI-чек «Dockerfile.dev и Dockerfile.production имеют совпадающий блок apt-install» (grep/diff в GitHub Actions).
