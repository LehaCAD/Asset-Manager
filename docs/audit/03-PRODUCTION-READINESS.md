> Что нужно доработать для коммерческого запуска. Приоритизированный план.

## Текущий статус

**Раскадровка** — технически зрелый MVP. Ядро работает: генерация, организация, шеринг, кредиты. Но для **коммерческого запуска с платящими пользователями** есть ряд критических пробелов.

**Общая готовность к запуску: ~60%**

| Область | Готовность | Блокер для запуска? |
|---------|-----------|-------------------|
| Генерация AI | 90% | Нет |
| Организация контента | 85% | Нет |
| Шеринг и ревью | 80% | Нет |
| Кредитная система | 95% | Нет |
| Аутентификация | 80% | Нет |
| **Оплата и тарифы** | **0%** | **Да — без денег нет бизнеса** |
| **Onboarding** | **0%** | **Да — пользователь не поймёт продукт** |
| Мониторинг и ops | 30% | Да — не увидишь проблемы |
| Безопасность | 60% | Частично |
| Кабинет | 50% | Нет (но выглядит сыро) |
| Мобильная версия | 20% | Нет (но теряешь аудиторию) |

## Блокеры запуска (без этого нельзя брать деньги)

### 1. Платёжная интеграция

**Статус:** Не реализовано вообще.

**Что нужно:**
- Интеграция с ЮKassa (самый простой вариант для РФ, комиссия ~3.5%)
- Страница пополнения баланса в кабинете
- Пакеты кадров: выбор суммы → оплата → начисление на баланс
- Автоматическое пополнение (опционально, для продвинутых пользователей)
- Чеки по 54-ФЗ (ЮKassa решает через свой кассовый модуль)
- Webhook от платёжки → начисление кредитов
- История платежей

**Оценка трудозатрат:** 2-3 недели (backend + frontend + тестирование + юридика)

**Почему ЮKassa:**
- Работает в РФ без проблем
- Поддерживает все карты, СБП, электронные кошельки
- Берёт на себя 54-ФЗ (онлайн-кассу)
- API хорошо документирован
- Минимальная комиссия для стартапа

4. Хранить секреты в `.env` только на сервере, не в репозитории

**Оценка:** 2-3 часа

## Высокий приоритет (первый месяц после запуска)

### 5. Мониторинг и алертинг

**Текущее состояние:** Sentry ловит ошибки (10% sampling). Больше ничего.

**Минимальный набор:**
- **Uptime monitoring:** простейший — UptimeRobot или аналог на /api/health/
- **Celery мониторинг:** Flower (веб-интерфейс для Celery) — видеть очередь, failed задачи
- **Логи:** centralized logging (хотя бы Docker logging driver → файл + logrotate)
- **Алерты:** email/Telegram при даунтайме, failed задачах, ошибках Sentry

**Продвинутый набор (когда пойдёт трафик):**
- Prometheus + Grafana для метрик
- Structured JSON logging
- APM (Application Performance Monitoring)

**Оценка:** 1 неделя (минимум), 3 недели (продвинутый)

---

### 6. Remote backups

**Текущее состояние:** pg_dump раз в день, хранится на том же VPS.

**Что нужно:**
- Копирование бэкапов в S3 (отдельный бакет или Backblaze B2)
- Проверка целостности бэкапа (автоматический тестовый restore)
- Алерт если бэкап не создался
- Redis persistence (AOF)

**Оценка:** 2-3 дня

## Что сейчас отлично и не трогать

Эти блоки уже production-ready. Не нужно рефакторить "для красоты" — они работают:

1. **Кредитная система** — atomic, auditable, idempotent. Не трогать.
2. **AI-генерация flow** — webhook + polling, optimistic UI, schema-driven. Не трогать.
3. **Element grid** — DnD, multi-select, bulk actions, filtering. Не трогать (только расширять).
4. **WebSocket real-time** — reconnect, JWT auth, event handling. Не трогать.
5. **Presigned upload** — двухфазная загрузка через S3. Не трогать.
6. **Design system** — oklch tokens, dark/light, shadcn/ui. Не трогать (только дополнять).

## Риски при запуске

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| **Kie.ai падает** | Средняя | Высокое — вся генерация стоит | Подключить второго провайдера (response_mapping это позволяет) |
| **DDoS** | Низкая | Высокое | Cloudflare Free plan + nginx rate limiting |
| **Исчерпание Celery** | Средняя | Среднее — очередь растёт, пользователи ждут | Мониторинг очереди + автоскейлинг workers |
| **S3 Timeweb down** | Низкая | Высокое — весь контент недоступен | CDN перед S3, кеширование thumbnails |
| **Спам комментариев** | Средняя | Низкое | CAPTCHA на публичных формах |
| **SQL injection** | Минимальная | Высокое | Django ORM защищает, но нужен security audit |
| **Утечка данных** | Средняя (секреты в git) | Критическое | Немедленная ротация ключей |

## Метрики для отслеживания после запуска

### Product metrics
- **Time to first generation** — сколько времени от регистрации до первой генерации
- **Generation success rate** — % успешных генераций (цель: >95%)
- **Share link creation rate** — какой % пользователей создаёт ссылки для ревью
- **DAU / WAU / MAU** — активность
- **Retention D1/D7/D30** — возвращаемость

### Business metrics
- **Conversion rate** — регистрация → первая оплата
- **ARPU** — средний доход на пользователя
- **Churn rate** — отток
- **LTV** — lifetime value
- **CAC** — стоимость привлечения

### Technical metrics
- **API response time p95** — цель: <500ms
- **Generation time p95** — зависит от провайдера
- **Error rate** — цель: <1%
- **Uptime** — цель: 99.5%
- **Celery queue depth** — цель: <50 задач

## Заключение

**Раскадровка готова к запуску как продукт, но не готова как бизнес.**

Техническое ядро — генерация, организация, шеринг, кредиты — это 80% работы, и она сделана хорошо. Оставшиеся 20% — это платёжка, onboarding, мониторинг и полировка — но именно они превращают проект в бизнес.

**Реалистичный срок до коммерческого запуска: 4-6 недель** при фокусе на:
1. Секреты и безопасность (неделя 0)
2. Платёжка + тарифы (недели 1-2)
3. Onboarding + error handling (неделя 3)
4. Кабинет + мобильная адаптация (недели 4-5)
5. Тестирование и soft launch (неделя 6)

Подробности о тарифах и юнит-экономике — в документе `02-BUSINESS-MODEL.md`.
