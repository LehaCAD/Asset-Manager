# Блок 4 — Элементы, метаданные, статусы и шеринг 2.0

> Большой продуктовый блок. Сначала ресёрч и проектирование, потом реализация. Все задачи связаны: какие данные показывать на элементах → как фильтровать → как шерить.

## Почему это один блок

Метаданные элементов, статусы, вкладка "Вид" и шеринг — одна цепочка:
- Какие **метаданные** есть у элемента (источник, статус, рейтинг) → определяет что показывать
- **Вкладка "Вид"** управляет отображением этих метаданных
- **Статусы** (готово / в работе / требует согласования) → основа для фильтрации при шеринге
- **Шеринг** использует всё это для командной работы менеджер ↔ криейтор

Проектировать по отдельности — потом переделывать.

---

## Часть A: Метаданные и отображение элементов

### 1. Какие метаданные показывать на карточке элемента

Нужно продумать:
- **Звёздочки (рейтинг):** обязательно ли показывать пустые? Сейчас неопрятно.
- **Тип элемента:** иконка типа (изображение/видео) — показывать.
- **Источник:** загружено извне или сгенерировано в сервисе — визуально различать.
- **Статус:** готово / в работе / требует согласования / на согласовании. Потом нужно для фильтрации и шеринга.

### 2. Вкладка "Вид" — переработка

- Сменить иконку с глазика на что-то, передающее ресайз / изменение aspect ratio.
- Добавить переключатель: показывать описание элементов или нет.
- Описание карточки: название, размер, дата добавления, стоимость генерации.

## Часть B: Шеринг 2.0

### Контекст

Шеринг и комментарии — ключевая фича для команд. Раскадровка работает именно для команд (менеджер + криейтор). Нужно вникнуть в кейсы и спроектировать так, чтобы фича вызывала желание платить.

### Вопросы для проработки

- Когда шеришь ссылку — можно ли туда добавлять элементы после создания?
- Быстрое добавление элементов со статусом "требует согласования" в ссылку?
- Какие ещё сценарии нужны менеджеру? Криейтору?
- Шеринг на уровне группы? Проекта?
- Как сделать так, чтобы за это хотелось платить?

### Целевые персоны

- **Менеджер:** создаёт ссылки для согласования, отслеживает статусы, оставляет комментарии
- **Криейтор:** получает ссылку, видит что нужно переделать, отмечает готовность

## Подход

1. Ресёрч конкурентов (Frame.io, Wipster, Filestage — review workflow)
2. Спроектировать модель метаданных элемента (какие поля, какие значения)
3. Описать user stories для менеджера и криейтора
4. Нарисовать mockup в pen: карточка элемента с метаданными, вкладка "Вид", шеринг flow
5. Согласовать
6. Составить техплан (API + frontend)
7. Реализовать
