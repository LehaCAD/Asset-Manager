# Багфикс-сессии — продакшн-тест 2026-04-18

Источник: сессия ручного тестирования с двух аккаунтов (iPhone + десктоп) на `raskadrawka.ru`.
Все замечания пользователя сохранены дословно (перефразировка только для сжатия).
Каждый баг помечен идентификатором `BF-XX`, чтобы на него можно было ссылаться из PR и коммитов.

1. Прочитать этот README целиком.
2. Открыть карту сессий ниже → выбрать ближайшую незакрытую строчку с приоритетом P0, если пользователь не попросил конкретную.

## Регламент работы (обязательный)

### 1. Где править код

**Все правки — только в локальном репо** (`C:\Users\ImpressivePC\Desktop\Asset Manager main`, Windows-хост).
Редактировать код напрямую на сервере (`/root/Asset-Manager`) **запрещено**:
- Сервер — чистый чекаут, синхронизируется через `git pull`.
- Прямая правка на сервере разойдётся с локалкой, следующий `git pull` создаст конфликт или потеряет работу.
- Исключение — правка `.env` на сервере (он в `.gitignore`). Но в рамках багфикс-сессий `.env` обычно не трогаем.

### 2. Проактивная ревизия смежных мест

Каждая правка — повод посмотреть влево-вправо:
- поправил цвет в dark → проверь light, и наоборот;
- трогал тост → грепни все точки вызова;
- правил мобилку → убедись, десктоп не сломался;
- нашёл антипаттерн в одном месте → грепни по всему коду, нет ли того же в других.

### 3. Светлая/тёмная тема

Большая часть UI-проблем идёт отсюда. После любого визуального фикса — скрин в обеих темах, сравнение.

### 4. Захардкоженные цвета — баг

`bg-[#xxxxxx]` или `text-white/black` вне сценария с гарантированным фоном — кандидат на замену на токен из дизайн-системы (`--background`, `--foreground`, `--muted`, и т.д.; см. `docs/BRANDBOOK.md`, `pencil-new.pen` Color System v2 node `gx1cz`).

### 5. Актуализация документации — обязательна при значимых изменениях

Значимое = новая модель / новый API-эндпоинт / изменение бизнес-правила / перенос файла / изменение контракта с фронтом / новый WS-event / изменение правил темы / новый компонент дизайн-системы.

В таком случае **в той же сессии**:
- обновить соответствующий `docs/systems/X.md` (маппинг доменов — в `CLAUDE.md` и `docs/MODULE_MAP.md`);
- если `X.md` отсутствует — создать заготовку;
- если правка касается дизайн-системы — обновить `pen/pencil-new.pen` (через Pencil MCP) + `docs/BRANDBOOK.md`;
- если затронут инвентарь компонентов — обновить `docs/COMPONENT_INVENTORY.md`.

**Незначимое** (фикс бага, рефактор без смены интерфейса) — доки трогать не надо.

### 6. Перед правкой — прочитать контекст

- `CLAUDE.md` (в корне проекта) — правила сессии и архитектуры.
- `docs/systems/X.md` для домена, в котором работаем (если есть ✓).
- Если `X.md` ⚠ не написан — `docs/MODULE_MAP.md`.
- `docs/ARCHITECTURE_CONSTITUTION.md` — чеклисты ревью, протоколы дебага.

### 7. Коммит — после подтверждения результата

Порядок (ни при каких обстоятельствах не коммитить преждевременно):

1. Сделать правки по `BF-XX` из сессии.
2. Обновить соответствующие `docs/systems/*.md`, если правка значимая (п.5).
3. Отметить `BF-XX` как ✅ в файле сессии, описав коротко фикс и что проверено смежно.
4. **Отдать результат пользователю на проверку.** Пользователь запускает прод-смоук.
5. **Только после «ок» от пользователя** — `git add <конкретные файлы>` + `git commit`.
6. `git push` — после коммита. Если пользователь не просил пушить, не пушим.

**Никогда не коммитить без явного «ок»** — даже если «всё же работает, код чистый, тесты зелёные». Если пользователь ушёл из сессии без подтверждения — оставить правки в working tree и в отчёте явно указать «ждёт подтверждения для коммита».

### 8. Деплой

После коммита и пуша:
```bash
ssh myvps
cd /root/Asset-Manager
git pull
docker compose -f docker-compose.production.yml up -d --build <services>
docker compose -f docker-compose.production.yml restart nginx   # если правился nginx.conf
```

- Менялся только бэк-код → `--build backend celery`.
- Менялся frontend → `--build frontend` (занимает ~15 минут, делаем реже, батчим правки).
- Менялся `Dockerfile.production` / `requirements.txt` → обязательно `--build`.
- Менялись миграции Django → `docker compose exec backend python manage.py migrate` после `up`.

### 9. Финальный отчёт по сессии

- Список закрытых `BF-XX` с кратким описанием фикса каждого.
- Список «смежно проверено, не трогал».
- Список «нашёл новое, завёл `BF-XX` в сессии Y».
- Список «осталось в сессии» с причиной (нужен доступ, нужен дизайн, перенесли в бэклог).

### 10. Не накатывать правки поштучно

В рамках одной сессии — один коммит / один PR / один ревью-цикл. Исключения — P0-блокеры, которые нужно немедленно в прод.

## Порядок исполнения

Рекомендую сначала **P0** (диагностика + поломанные вещи), параллельно начинать **01 темы** и **02 тосты** — это затрагивает десятки компонентов, займёт больше всего времени.

**Сессии можно пускать параллельно при условии непересекающихся файлов.** Кандидаты на параллельный старт:
- 00 (диагностика) + 07 (libmagic) — чисто бэкенд, не трогают UI
- 05 (ачивки) + 06 (realtime) — трогают onboarding/notifications/credits-store, частично пересекаются — сначала 05 смержить
- 03 (модалки) + 08 (шеринг) — разные модальные стеки

**Пересекаются, нельзя параллелить:**
- 01 (темы) с любой UI-сессией — 01 первой, остальные на актуализированной базе
- 02 (тосты) с 05/08 — они правят вызовы тостов
