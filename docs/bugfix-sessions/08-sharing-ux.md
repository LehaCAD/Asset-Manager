## BF-08-02 — Непредсказуемое поведение «Создать и скопировать» на мобилке ✅

**Симптом:**
> «Нажимаешь „создать и скопировать" — в мобилке не видно, что дальше. Ничего не произошло. Нажимаешь ещё раз — сверху тост „не удалось создать ссылку". Потом заходишь в отзывы — на самом деле создалась. Непредсказуемое поведение. После нажатия модалка должна закрыться».

**Корень (подтверждён чтением кода):**
В `CreateLinkDialog.handleSubmit` создание ссылки и `navigator.clipboard.writeText` были в одном `try`. На iOS Safari после `await createLink` теряется user gesture — `clipboard.writeText` падает `NotAllowedError`. Единый catch кричал «Не удалось создать ссылку», модалка не закрывалась, хотя ссылка уже лежала в БД.

**Фикс (`frontend/components/sharing/CreateLinkDialog.tsx:87-120`):**
1. Создание ссылки — отдельный try/catch. Ошибка сети → единственный error-тост, выход.
2. После успешного POST модалка закрывается в `finally` (независимо от судьбы clipboard).
3. Clipboard — отдельный try/catch. Успех → «Ссылка скопирована». Фейл → «Ссылка создана — скопируйте её вручную» (юзер видит ссылку в списке отзывов/ShareLinksPanel).
4. Guard `if (isSubmitting) return` + существующий `disabled={isSubmitting}` исключают повторный POST.

**Смежно проверено:** `ShareLinksPanel.handleCopy` уже показывает свой тост «Ссылка скопирована» при клике по кнопке в списке — конфликта нет, т.к. пути разные.

## BF-08-04 — Баг из бэклога: реакции в шеринг-лайтбоксе не ставятся ⏳

**Источник:** `docs/Фиксы и мысли.md` пункт 9. Память `project_sharing_bugs.md` — «reaction counter spam on fast clicks».

**Проверка кода (ничего очевидно битого не нашёл):**
- `app/share/[token]/page.tsx:870-911` — `handleCardReaction` закрыт `reactingRef: Set<number>` → повторный клик по тому же элементу блокируется до ответа сервера (анти-спам счётчика).
- `ReviewerLightbox.tsx:172-179` — `handleReaction` требует `sessionId`, иначе info-тост; дальше пробрасывает в тот же `handleCardReaction`.
- Оптимистичный апдейт — только подсветка кнопки; счётчики берутся из `res.data.likes/dislikes` (source of truth — сервер).
- WS-канал принимает `reaction_updated` и мержит чужие клики в `project.scenes/ungrouped_elements`.

**Решение:** оставляю `BF-08-04` открытым до ручного воспроизведения пользователем. Если симптом «не ставится» сохраняется после деплоя фикса BF-08-02 — нужны шаги репро (устройство, тёмная/светлая, гость/автор ссылки), логи из Sentry и payload POST `/public/{token}/reactions/`. По статике кода проблема не находится.

## BF-08-05 — Проактивный обход шеринга end-to-end ⏳

**Задача:** пройти весь flow:
1. Создать ссылку для группы.
2. Скопировать.
3. Открыть в incognito на другом устройстве (симулирует ревьюера).
4. Полистать лайтбокс, оставить комментарий, поставить реакцию, проставить статус (согласовано/требует правок).
5. Посмотреть как это отражается у создателя.

**Статус:** требует живого прода (прод-смоук пользователем). Из агента автономно не выполнимо. Делается после деплоя BF-08-02, параллельно с BF-08-04 — если в e2e всплывает «реакция не поставилась» — фиксируем стек и возвращаемся.
